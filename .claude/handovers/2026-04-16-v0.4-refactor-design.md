## Bug fixes (second session)

After the design doc was finalized, Chris directed Claude to empirically verify the §3 methodology-equivalence claim rather than defer to coauthors. Claude:

1. **Read DMSS (2024) Appendix C equation (77)** and DMQSS (2026) Appendix B to confirm what object d(markup)/d(θ) and the weight matrix should be.
2. **Wrote `tests/test_first_stage_correction.py`** with three test classes: `compute_delta` shares-restoration test, end-to-end TRV/F equivalence, isolated markup-gradient equivalence.
3. **Ran the tests** and found three real bugs, all in code added yesterday as part of CClean-fixes `demand_params` feature (except bug #3 which is pre-existing).

### The three bugs

**Bug 1 (sign error) — `problem.py:1045`.**
```python
# Before:  gradient_markups[m][idx, 0] = mu_t / alpha
# After:   gradient_markups[m][idx, 0] = -mu_t / alpha
```
Correct formula is −markup/α per homogeneity of degree −1 in D. Caught by `test_gradient_markups_match`. Full derivation in the commit's docstring.

**Bug 2 (missing concentration adjustment) — `_compute_analytical_demand_adjustment`.**
The analytic path was computing H = (1/N) Z' ∂ξ/∂θ directly without the 2SLS residualization on X_D that the PyBLP path has always applied. Ported the concentration block (~10 lines) so β is correctly concentrated out per DMSS's "linear parameters profiled" convention.

**Bug 3 (wrong weight matrix, pre-existing) — `_compute_demand_adjustment_gradient`.**
The PyBLP path was using `demand_results.updated_W` (the "next-step" efficient weight that would be used for a hypothetical 2-step update) instead of `demand_results.W` (the weight *actually used* in estimation). DMSS eq (77) specifies the weight used in estimation. This bug is pre-CClean-fixes; all prior users of the PyBLP path with `demand_adjustment=True` got slightly-wrong TRV/F.

### Backwards-compat flag

Added `pyRVtest.options.demand_adjustment_weight`:
- `'W'` (default, v0.3.3+): DMSS-correct.
- `'updated_W'`: reproduces pre-v0.3.3 PyBLP-path behavior for replication.

Set to `'updated_W'` before `.solve()` to reproduce prior output. Documented in `options.py` docstring.

### Verification

After all three fixes on the same DGP:
- markups match to machine precision (1e-8)
- g, H, H_prime_wd, h_i all match to machine precision (< 1e-14)
- TRV with `demand_adjustment=True`: r1 = −0.16875467 vs r2 = −0.16875468 (diff 1e-8)
