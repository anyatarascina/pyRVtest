## 3. Add the `demand_adjustment + endogenous_cost_component` gate

**Status (v0.4):** the combination is silently allowed and produces biased
demand-adjusted variance. `_compute_demand_adjustment_gradient` captures
`d markups_effective[m] / d theta` but misses the `d gamma_m / d theta`
channel that flows through the IV correction.

**Where this surfaced:** the regression test
[tests/regressions/test_memo1_section42_adjustment_interaction.py](../../tests/regressions/test_memo1_section42_adjustment_interaction.py)
expects `ValueError`; currently marked `xfail(strict=True)` with a reason
pointing here. When the gate lands, the test flips to XPASS and the marker
must be removed.

**v0.5 options:**
- **Option a (short-term):** add the validation that raises `ValueError`
  when both flags are set. Honest about the limitation, low effort.
- **Option b (proper):** extend `_compute_demand_adjustment_gradient` to
  re-run `_compute_iv_correction` inside each finite-difference perturbation
  so the derivative captures `dgamma/dtheta`. More work; needed eventually
  if users want to combine these features.

Pick (a) for v0.5.0; revisit (b) later only if a user requests it.

## 5. F-stat collapse under identically-zero-markup candidate models

**Symptom:** in the README quick-start output (`pyRVtest.data.load_example`,
PC truth, 4 candidate models including PerfectCompetition), every F-stat
cell involving model 3 (PC) is suspiciously small — F = 0.0, 1.4, or 2.6
— even though pairwise TRVs involving PC are sharply significant
(``***``). For example, the (Bertrand, PC) pair shows ``TRV = 6.771***``
but ``F = 2.6``, and the (Cournot, PC) pair shows ``TRV = 0.419`` (the
Dearing-degenerate cell, expected) with ``F = 0.0``.

**Hypothesis:** when one model in a pair has identically-zero markup
(``PerfectCompetition``), the residual variance for that model collapses,
the σ-cross terms vanish, and the F denominator structure degenerates
even when the moment-condition difference is well-defined. The TRV
statistic and MCS p-values look correct in this regime; only the F-stat
diagnostic appears off.

**Where to look:** F-stat computation in
`pyRVtest/solve/test_engine.py` around the σ̂_0, σ̂_1, σ̂_2 trace
contractions and the F numerator/denominator assembly (search for
``F_numerator`` and ``F_denominator``).

**Decision needed:** is this a real bug to fix, or an inherent property
of the F-stat formula at the zero-markup boundary that should be
documented as a known limitation? The TRV / MCS outputs are clean, so
the F-stat is the only signal that misleads — possible to live with
either way.

**Discovered:** 2026-05-05 docs Phase 1 session, while writing the
README quick-start narrative for the shipped synthetic example.

## 6. Rewrite notebook tutorials to v0.4 class-based API + re-execute

**Symptom:** `docs/notebooks/testing_firm_conduct.ipynb` and
`docs/notebooks/model_library.ipynb` use the v0.3 string-based
`ModelFormulation(model_downstream='bertrand', ...)` API throughout
(17 and 31 hits respectively). A user following the notebooks learns
the deprecated API and sees `DeprecationWarning` on every cell that
constructs a candidate model. v0.4 added a deprecation banner cell at
the top of each notebook (commit landing 2026-05-06) explaining the
state and pointing users at the v0.4 walkthrough in `docs/tutorial.rst`.

**Why deferred:** rewriting the input cells to the class-based API
without re-executing the notebooks would leave the saved outputs
showing results from `ModelFormulation` runs, which is worse than the
current state (the saved outputs would no longer match the input
code). Re-executing requires a working `pyblp` environment with the
right NumPy / pyblp pin combination — see `docs/faq.rst` on the
`numpy<2 + pyblp<1.2` vs `numpy>=2 + pyblp>=1.2` constraint. The
current dev machine pin combination does not run pyblp.

**v0.5 work:**
1. Rewrite all `pyRVtest.ModelFormulation(model_downstream='X', ...)`
   constructions to `pyRVtest.<X-class>(ownership=...)` in both
   notebooks. The trickier cases:
   - Vertical models with both `model_downstream` and `model_upstream`
     → `pyRVtest.Vertical(downstream=..., upstream=..., ...)`.
   - Per-model `unit_tax='col'` / `advalorem_tax='col'` → move to
     `pyRVtest.Problem(unit_tax='col', advalorem_tax='col', advalorem_payer='consumer', ...)`.
   - `kappa_specification_downstream` → `pyRVtest.PartialCollusion(...)`.
   - `user_supplied_markups='col'` → `pyRVtest.UserSuppliedMarkups('col')`.
   - `custom_model_specification=callable` → `pyRVtest.CustomConductModel(callable)`.
2. Update `model_formulations=` keyword to `models=`.
3. Re-execute end-to-end with a working `pyblp` install so the saved
   outputs match the new code.
4. Remove the deprecation banner cell once the modernization is done.
