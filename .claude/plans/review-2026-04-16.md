# Review of v0.4 Refactor Plan — Hard-Push Edition

## Context

Chris asked for a full review (gaps + sequencing + readiness) of [.claude/plans/v0.4-refactor.md](pyrvtest/.claude/plans/v0.4-refactor.md) with adversarial push-back. The plan was produced across two sessions on 2026-04-16 by an earlier Claude that (by its own admission) **flipped three times on scope** during the drafting. That pattern is a real honesty-flag and motivated a critical re-read.

This review is written to be executable: each finding has a proposed fix or decision the plan should capture before Step 0 begins. I am not recommending the plan be scrapped — the core motivation (monolithic `problem.py`, PyBLP coupling, dual demand paths) is sound and the prototype work is real. I am recommending that several sequencing choices, scope decisions, and readiness gaps be resolved before any branching from `CClean-fixes`.

## I. Execution-readiness issues (blocking Step 0)

### 1. b3b08a3 is unpushed and coauthors are running broken code

**Evidence.** [.claude/handovers/2026-04-16-v0.4-refactor-design.md:5](pyrvtest/.claude/handovers/2026-04-16-v0.4-refactor-design.md:5) says "`b3b08a3` (not yet pushed)". Remote `CClean-fixes` therefore lacks the three first-stage-correction fixes.

**Why this blocks.** Marco and Lorenzo, if they pull `CClean-fixes` today, get the pre-fix state: wrong sign in analytical `d(markup)/dα`, missing concentration adjustment, `updated_W` instead of `W`. Any replication they run produces wrong TRV/F. Every day this is unpushed, they may be accumulating incorrect results.

**Fix.** Push `b3b08a3` **before** doing any other work. Send [MEMO_coauthor_updates.md](pyrvtest/MEMO_coauthor_updates.md) to Marco and Lorenzo the same day. Ask them to confirm receipt.

### 2. The `v0.3.2-stable` anchor tag is placed wrong

**Evidence.** [Design doc §5 line 765](pyrvtest/.claude/plans/v0.4-refactor.md:765) says `git tag -a v0.3.2-stable 45bcc4b`. But `45bcc4b` is three commits before the b3b08a3 bug fixes. The "stable anchor" would therefore point to a version with three known bugs in it.

**Why this is a problem.** If v0.4 fails and we invoke the nuclear option, we'd reset to a state where `demand_adjustment=True` gives wrong answers. That's not a stable point; it's a broken point.

**Fix.** Move the tag to `b3b08a3` (or whatever the final CClean-fixes commit is after Step 0 items 0b/0d/0e/0f land). Rename tag to `v0.3.3-stable` to reflect that the bug fixes are in the baseline. Update §5 0a and §10 accordingly.

### 3. Step 0c's "DONE" status is misleading

**Evidence.** [§5 line 767](pyrvtest/.claude/plans/v0.4-refactor.md:767) marks 0c as "DONE (as of commit `b3b08a3`)". [tests/test_first_stage_correction.py](pyrvtest/tests/test_first_stage_correction.py) exists (471 lines, 9 tests).

**What 0c actually protects against.** Regression in the *specific* PyBLP-vs-demand_params equivalence on *one* DGP. That's narrow.

**What 0c does NOT protect against.** It does not snapshot markups, TRV, F, MCS across the seven existing test files. It does not catch a bug in `Bertrand.compute_markup` that silently changes test outputs by 1e-6. That's what 0b (snapshot suite) is for — and 0b is still pending.

**Fix.** Leave 0c marked DONE but add a prominent note in §5 that **Step 0 as a whole is not complete until 0a, 0b, 0d, 0e, 0f also land on CClean-fixes**. The current wording makes it read as "most of Step 0 is done" when actually the safety net has a hole big enough to walk through.

### 4. No definition of "Step 0 done"

**Evidence.** §5 lists six sub-tasks (0a–0f) but no acceptance criterion. When do we branch `v0.4-refactor` off `CClean-fixes`?

**Concrete ambiguity.** Does Step 0 end when:
- (a) All six tasks are committed to CClean-fixes? (File-count criterion)
- (b) All Step 0 tests pass in CI? (There is no CI — see §III.6)
- (c) Marco and Lorenzo have reviewed 0b and 0d? (Coauthor gate)
- (d) The DMSS yogurt replication matches published values to a specified precision? (Empirical gate)

**Fix.** Add an "Acceptance criteria for Step 0" block under §5. My recommendation: **all of a + b + d**. Coauthor review (c) is desirable but not blocking — the golden file (d) is the empirical gate, and if Lorenzo can't reproduce the published DMSS yogurt numbers with pyRVtest, the math is wrong regardless of whether he's formally signed off.

## II. Structural sequencing issues

### 1. Dearing notation verification (step 12) ships AFTER model classes (step 5). This is backwards.

**Evidence.** [§7 Open Question 1](pyrvtest/.claude/plans/v0.4-refactor.md:820): "Verify against `LearningFirmConduct.pdf` before writing model classes (step 12 of migration). Working assumption: constant_markup Δ = m̄, rule_of_thumb Δ = λ·p, cost_plus Δ = λ·c."

The working assumption is not verified. Yet step 5 ships `RuleOfThumb`, `ConstantMarkup`, `CostPlus` with those specific formulas ([§4.2 lines 519–565](pyrvtest/.claude/plans/v0.4-refactor.md:519)). Step 12 then reads the Dearing paper to check.

**Push.** This is a plain ordering mistake. If step 12 finds the parameterization is wrong, step 5's classes are already in the main branch, all the backward-compat aliases in `ModelFormulation._registry` point to them, any user who adopts `RuleOfThumb(markup_rate=0.3)` in the interim gets a garbage API, and the fix requires either a breaking change or a second deprecation alias layered on top of the first.

**Fix.** Either:
- (a) Move Dearing verification to a prerequisite of step 5 (new step 0.5 or an extension to step 0), OR
- (b) Ship `Bertrand`, `Cournot`, `Monopoly`, `PerfectCompetition`, `MixCournotBertrand`, `PartialCollusion` in step 5 and defer `RuleOfThumb`/`ConstantMarkup`/`CostPlus` to the merged step 5+12 later.

I lean (b) because it decouples the refactor-mechanical work (most of step 5) from the paper-notation work (step 12).

### 2. Property tests (step 20) are positioned far too late

**Evidence.** §5 step 20 is the Hypothesis-based property test suite. It lives between "error messages" (19) and "doctest" (21). Meanwhile the handover's own retrospective notes — see the "Anti-sycophancy note for the next Claude" — doesn't surface this. But the Explore agent's critique (which I verified independently) does.

**Push.** Property tests like determinism, row-permutation invariance, market-partition invariance, and FWL identity are general invariants of the testing engine. They should run against every commit after step 3 (backend refactor) and step 4 (demand adjustment unification) — the steps most likely to introduce silent regressions in those invariants. Scheduling them at step 20 means 16 architectural steps will have merged without these guardrails.

**Fix.** Write a minimal Hypothesis suite as part of step 0f (scaffolding exists; actually populate it with 3–5 core properties). Add the full property suite incrementally as step 4, 7, 8 land. Step 20 becomes "final audit and gap-filling," not "first implementation."

### 3. `mypy --strict` at step 17 is too late to prevent design mistakes

**Evidence.** §5 step 17: "`mypy --strict` pass across entire package. Fix all type issues." That's after 16 steps of architectural work.

**Push.** Type checking's value is catching design errors early. If the `DemandBackend` protocol has inconsistent signatures between `PyBLPBackend` and `LogitBackend`, mypy catches this the moment they're written. Deferring mypy to step 17 means you'd find such issues only after 15 commits of downstream code has built on the broken contract.

**Fix.** Run mypy starting at step 1. The target at step 17 should be "mypy --strict on 100% of the package" — but partial strictness (mypy on `backends/`, then `backends/ + models/`, etc.) should be enforced incrementally as each new module lands. This is cheap if done per-commit; expensive if left for the end.

### 4. AFSSZ prototype (step 16) has ambiguous blocking status

**Evidence.** §5 step 16: "Prototype AFSSZ wrapper (3-5 market-years, fake data) ... Iterate on pyRVtest API for any pain points." §9 says "Not a commitment to a timeline." No explicit statement of whether v0.4 releases with or without AFSSZ integration validated.

**Push.** This matters because AFSSZ scalable testing is one of the three motivating use cases in §1. If v0.4 ships and AFSSZ integration then reveals the API is awkward, we need a v0.5 to fix it — defeating the "coauthor migrates once" principle in §8 ([line 844](pyrvtest/.claude/plans/v0.4-refactor.md:844)).

**Fix.** Decide explicitly: is step 16 a blocker for v0.4 release (step 25)? My recommendation: **yes, it's a blocker**. The prototype should surface API pain points *before* v0.4 locks. If AFSSZ is blocked on other dependencies (data, PyBLP estimation), that's a v0.4 scheduling constraint that should be surfaced now.

### 5. Worked-example step (15) depends on labor project data that may not exist

**Evidence.** §7 Open Question 5: `UserSuppliedBackend` worked example target. Lean is "stylized Almagro-Sood-style ordered-choice labor supply." But §2 item 9 says `LaborSupplyBackend` in v0.4 is a *skeleton* because "full implementation when labor project has data" (§2 lines 66-68).

**Push.** If the real labor data isn't ready, the worked example is a guess. If the guess happens to differ from the real Almagro-Sood structure, we'll need to redo the worked example in v0.4.1 once data arrives — again defeating "migrate once."

**Fix.** Decide: is step 15 gated on real labor data being available? If yes, the whole v0.4 release timeline depends on the labor project. If no, accept that the worked example is illustrative not canonical, and write a note to that effect in `docs/custom_demand.md` ("This example is stylized; the actual Almagro-Sood integration will differ in ways X, Y, Z").

### 6. No CI in v0.4 — released on manual test execution

**Evidence.** [§2 Defer to v0.5+](pyrvtest/.claude/plans/v0.4-refactor.md:72): "GitHub Actions CI (depends on 16-17 being stable first)."

**Push.** A package that depends on PyBLP internals, ships to PyPI, and has no CI is one PyBLP rename away from silent breakage. The refactor's whole motivation includes "fragile PyBLP internal coupling" (§1). Deferring CI to v0.5 keeps the fragility in place for the entire v0.4 lifetime.

**Fix.** Move CI into v0.4 scope (new step 24.5 or augment step 22/25). A minimal workflow is: `pip install -e .`, run pytest, run mypy --strict, run doctest. Maybe 20 lines of YAML. The dependency claim ("depends on 16-17 being stable first") is true in the sense that CI will be red if 17 isn't clean — but it's false as a sequencing argument because you want CI red for known-broken code, not absent.

### 7. Logging and error messages (18, 19) can interleave with earlier steps

**Evidence.** §5 steps 18 and 19 come after the architectural refactor (1–16). They're positioned as cross-cutting cleanup.

**Push.** Not really a problem as long as it's acknowledged these are cosmetic for internal refactor correctness. They don't change behavior. But the plan should say so explicitly, because otherwise they look like critical-path work.

**Fix.** Annotate steps 18 and 19 as "may run in parallel with any earlier step once scaffolding is in place." They need not sequence after 17. No change to critical path.

## III. Critical gaps and ambiguities

### 1. No formal coauthor-review checkpoints

**Evidence.** [§9](pyrvtest/.claude/plans/v0.4-refactor.md:864): "Marco/Lorenzo should sanity-check before unifying the paths in step 4." That's the only mentioned coauthor gate.

**Push.** The plan touches methodology (steps 4, 7, 12), labor theory (step 14), and external application (step 16). All of these benefit from coauthor sign-off. But none have formal gates. Without gates, coauthors may discover issues post-hoc in a review of a 25-commit branch rather than at natural pause points.

**Fix.** Add three gates to §5:
- **Gate A** (after step 4): Marco + Lorenzo review the unified demand adjustment against DMSS eq. (77) on the golden file. Block step 5 until signed off.
- **Gate B** (before step 14): Chris + (labor coauthor TBD) review sign conventions and labor model formulas against `basic_model.tex`. Block step 14 until signed off.
- **Gate C** (after step 16): Marco reviews AFSSZ integration. Block step 17 until signed off.

### 2. No quantitative snapshot-update decision rule

**Evidence.** §5 step 24: "Update `tests/snapshots/` with any deliberate v0.4 changes (e.g., if Hessian precision improved by 1e-7, snapshots are intentionally tightened)." §7 Open Question 9 elaborates.

**Push.** "Deliberate" is not operational. When step 7 lands and the Hessian changes from 1e-7 to 1e-15, is every downstream TRV now "deliberate"? What about step 4 (demand adjustment unification)? If the unified path produces TRV values that differ from snapshots by 1e-9 (plausibly just finite-difference step-size effects), is that "deliberate"?

**Fix.** Add a decision rule:
- Changes ≤ 1e-12 → snapshot update in same commit, note in CHANGELOG. Never triggers investigation.
- Changes 1e-12 to 1e-7 → snapshot update only if commit message identifies the source (e.g., "step 7 tightens Hessian precision, see §4.7"). Otherwise treat as regression.
- Changes > 1e-7 → block merge. Investigate root cause before updating snapshot.

### 3. No rollback-trigger criteria

**Evidence.** §10 lists *how* to roll back (soft/hard/nuclear). But not *when*.

**Push.** The rollback procedure is recovery infrastructure. It's useless if nobody pulls the trigger. When does "this step isn't quite working" become "revert and try again"? Today, the answer is implicit — maintainer judgment.

**Fix.** Add concrete triggers:
- **Soft revert trigger**: Single-step commit breaks any Step 0 test. Revert immediately, investigate, re-commit.
- **Hard revert trigger**: Two consecutive steps reveal issues with a step N that's now upstream. Reset to pre-N state.
- **Nuclear trigger**: Golden file (DMSS yogurt) drifts from published values by > 1% after any merge. Abandon v0.4-refactor branch; investigate what went wrong before restarting.

### 4. The "code quality is the constraint" principle is not operationalized

**Evidence.** [§8 line 844](pyrvtest/.claude/plans/v0.4-refactor.md:844): "Time is not the constraint; code quality is."

**Push.** "Code quality" is undefined. It's treated as a tie-breaker when decisions arise, but it has at least five plausible meanings (AI readability, type safety, test coverage, modularity, documentation). The plan includes items for all five, which is why it's large.

**Fix.** Either:
- (a) Pick an ordered definition: "code quality means (1) correctness verified by tests, (2) AI-navigability via clear module/class structure, (3) type safety, (4) modularity." Then items map to priority.
- (b) Accept that all five matter and acknowledge this expands scope. This is the current implicit position.

Either is valid. But the plan should state which.

## IV. Scope: honest independent reassessment

The prior Claude's "anti-sycophancy note" flags that it flipped three times on scope. I want to run an independent check: are all 25 items actually necessary, or did some get added by momentum rather than need?

**My position.** The core items (1–10) are well-motivated and load-bearing. Items 11–16 are real integrations supporting named use cases. **Items 17–24 are cross-cutting code-quality additions all added in one session in response to a single "is this the best code?" prompt.** Each is individually defensible; bundled they compound delay risk.

Specifically:

| Item | Category | Independent judgment |
|------|----------|---------------------|
| 1–4 (backend + pipeline) | Architecture | **Essential.** Solves the monolith + coupling + dual-path problems. |
| 5 (class-based models) | Architecture | **Strong.** But see §II.1 — timing issue with Dearing verification. |
| 6 (Dict_K, sigma→rho) | Bug fix + rename | **Essential.** Dict_K is a real bug; rename aligns with PyBLP. |
| 7 (analytical Hessian) | Correctness | **Nice-to-have, not essential.** Current 1e-7 precision hasn't caused observed issues. Could defer to v0.5 with low cost. |
| 8 (Problem.solve split) | Architecture | **Strong.** Completes the refactor story. |
| 9, 10 (results + panel) | Feature (AFSSZ) | **Strong.** Named use case. |
| 11 (build_passthrough) | Diagnostic feature | **Weak case.** Users can call the internal function. Could defer. |
| 12 (Dearing verification) | Correctness | **Essential — but should precede step 5.** |
| 13 (instrument helpers) | Convenience | **Weak case.** Researchers can build their own IVs. Could defer. |
| 14 (labor hooks) | Feature (labor project) | **Strong.** Named use case. |
| 15 (worked example) | Docs | **Strong.** AI-agent audience. |
| 16 (AFSSZ prototype) | Integration test | **Essential.** Must block release (see §II.4). |
| 17 (mypy --strict) | Quality | **Should be incremental, not a late step.** |
| 18 (logging) | Quality | **Weak case for v0.4.** Current `print`/`output` works. Defer. |
| 19 (rich errors) | Quality | **Nice-to-have.** Defer to v0.4.1. |
| 20 (property tests) | Correctness | **Should be incremental — see §II.2.** |
| 21 (doctest) | Docs + tests | **Strong.** AI-agent audience + cheap. |
| 22 (`__all__` pin) | API hygiene | **Cheap; ship with §5 step 1.** |
| 23 (`AGENTS.md` + guides) | Docs | **Strong.** Named design goal. |
| 24 (snapshot updates) | Mechanical | **Keep.** |
| 25 (tag + release) | Mechanical | **Keep.** |

**Proposed trimming.** A tighter v0.4 scope drops items 11, 13, 18, 19 to v0.4.1 (or later), and integrates items 17, 20, 22 into the incremental flow rather than as late steps. That leaves:
- Essential + strong: 1–10, 12, 14, 15, 16, 21, 23, 24, 25 (16 items)
- Incremental-throughout: 17, 20, 22 (not separate steps)
- Deferred: 7, 11, 13, 18, 19

I flag this as an option, not a recommendation. Chris has explicitly signed off on the maximalist scope ([§8 line 844](pyrvtest/.claude/plans/v0.4-refactor.md:844)) and there are real arguments for amortizing disruption. But the plan should at least acknowledge that this is a scope decision, not a constraint — so that if mid-migration reality bites (labor data delayed, AFSSZ surfaces major API problems, a coauthor leaves), the team knows which items are shedable.

**Fix.** Add a §2.5 "Scope tiers" block listing each item as Essential / Strong / Deferable. Document what gets cut first if pressure arises.

## V. Concerns that deserve explicit mention but don't rise to "blocking"

### 1. The methodology-equivalence claim rests on one DGP

**Evidence.** [§3 note 4 lines 362–373](pyrvtest/.claude/plans/v0.4-refactor.md:362) says both paths compute d(markup)/d(θ) to first order. The empirical verification after b3b08a3 shows r1 = −0.16875467 vs r2 = −0.16875468 on *one* DGP.

**Push.** Bug 3 (updated_W vs W) was pre-existing and lay dormant until the equivalence test. What if a different DGP (say, more markets, or with rho > 0 nesting, or with endogenous_cost_component) reveals a fourth latent bug? Step 0d (DMSS yogurt golden file) is designed to catch this — but until it runs, the equivalence claim is single-DGP-verified.

**Fix.** Explicitly defer the "methodology equivalence confirmed" claim's full confidence until Step 0d passes. §3 note 4 should be updated post-0d to say "confirmed on DGP X and golden file Y" rather than the current absolute phrasing.

### 2. NashBargaining's compute_markup raises NotImplementedError

**Evidence.** [§4.2 line 620](pyrvtest/.claude/plans/v0.4-refactor.md:620): `raise NotImplementedError("step 13 of migration")` inside `NashBargaining.compute_markup`. But step 13 is now instrument helpers in the current numbering; the labor models are in step 14. That's a stale internal reference inside the design doc.

**Push.** Minor but a symptom: the plan was rewritten and internal refs weren't fully re-synced.

**Fix.** Sweep design doc for step-number references inside code blocks. Ensure they match the current §5 numbering.

### 3. `problem.py` is 1,909 lines (plan says 1,855)

**Evidence.** `wc -l pyRVtest/problem.py` returns 1909. Plan says 1855 ([§1 line 15](pyrvtest/.claude/plans/v0.4-refactor.md:15)).

**Push.** Not a real issue — just means bug fixes added ~54 lines post-drafting. But it's a signal that the plan is a snapshot, not live state. When we actually start the refactor, this gap widens.

**Fix.** Update §1 numbers at Step 0 kickoff. Add a "snapshot taken YYYY-MM-DD" tag so future readers know the numbers are stale.

### 4. Backwards-compat deprecation warnings may flood user output

**Evidence.** Plan introduces deprecation warnings for `ModelFormulation` ([§4.2 line 663](pyrvtest/.claude/plans/v0.4-refactor.md:663)), `sigma` key in `demand_params` ([§6 line 810](pyrvtest/.claude/plans/v0.4-refactor.md:810)), possibly more. If a user has code that constructs 10 `ModelFormulation` objects and passes `demand_params={'sigma': [...]}`, they get 11+ warnings per run.

**Push.** The handover flags this as a concern ("deprecation-warning flood") surfaced by the 4.7 cross-model review. But the plan doesn't address it.

**Fix.** Use `warnings.warn(..., category=DeprecationWarning, stacklevel=2)` with `warnings.simplefilter('once', DeprecationWarning)` logic so each deprecation fires once per session. Add to §2 or §4.2 as a design note.

### 5. Labor sign-convention validation is missing

**Evidence.** Handover flags this as surfaced by 4.7 review but the plan doesn't encode it. [§4.5 lines 732–736](pyrvtest/.claude/plans/v0.4-refactor.md:732) says labor mode "validates presence of `wages`, `employment` columns" but doesn't validate signs.

**Push.** If a user passes labor data with `wages` as a negative number (sign error upstream) or with the sign convention of `compute_markup` mismatched to the pass-through direction, the test produces silent garbage.

**Fix.** Step 14 should add explicit sign-validation checks in `Problem.__init__` when `market_side='labor'`. Violations raise validation errors per step 19's rich-error pattern.

## IX. What this review is NOT

- Not a recommendation to scrap v0.4. The core refactor is sound.
- Not a commitment to a timeline. Each fix above is cheap individually.
- Not a rubber stamp. Several items are genuine course-corrections, not polish.
- Not exhaustive. I did not audit every line of the 892-line design doc. Focus was on motivation, migration plan (§5), open questions (§7), decisions log (§8), and rollback (§10). Prototype code (§3) and API details (§4.2–4.7) were read but not line-audited.
